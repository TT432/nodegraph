# TaskH 节点交互（客户端）

## 目标
在 `NodeGraphWidget` 画布上实现：左键拖节点头→移动节点；左键拖输出/输入端口→拉出连线（预览跟随鼠标、释放时按类型校验建立连接）；悬停 tooltip（TaskF 已交付，本任务确认其在本交互状态下仍生效）。所有结构性变更（移动、连接）走 TaskC 命令系统经 `UndoManager`，可撤销/重做。复用 TaskA（数据/连接约束）+ TaskC（命令）+ TaskE（事件路由/坐标变换）+ TaskF（NodeLayout 命中几何）+ TaskG（ConnectionDrag 预览渲染）。

## 前置
- TaskA：`NodeGraph`(node/nodes/canConnect/connect/inputConnection)、`Node`(id/x/y/setPosition)、`Port`、`ConnectResult`(OK/AUTO_CONVERTED/INCOMPATIBLE)。
- TaskC：`Command`、`UndoManager`(apply)、`MoveNodeCommand`(graph,target,newX,newY，首次 execute 捕获 old)、`ConnectCommand`(graph,fromNode,fromOutput,toNode,toInput)。
- TaskE：`Viewport`(worldToScreenX/Y, screenToWorldX/Y, scale)、`NodeGraphWidget`(getX/getY/width/height, isValidClickButton, 事件钩子)、`NodeGraphScreen`(事件转发契约)。
- TaskF：`NodeLayout`(headerContains/pickInputPort/pickOutputPort→Optional<Integer>/inputPort(i)/outputPort(i)→PortAnchor(x,y,index)/inputPortData/outputPortData→Port)。
- TaskG：`ConnectionDrag`、`ConnectionRenderer`(renderPreview/withAlpha/PREVIEW_ALPHA/WARN_COLOR)、`NodeGraphWidget.pending()/setPending(...)`。
- MC 1.20.1：AbstractWidget 事件链（mouseClicked/Dragged/Released，button==0 拖动默认可达；中键拖动经宿主转发 mouseMoved）。

## 关键设计决策

### D1 交互与渲染职责分离（NodeInteractionController）
新增纯交互控制器 `com.nodegraph.nodegraph.client.interaction.NodeInteractionController`，持有 `NodeGraphWidget` 引用（从中取 graph/viewport/undo/origin/getX/getY/setPending）。widget 的左键事件先委托 controller；controller 不接管时（未命中节点）widget 退回 PANNING 逻辑。控制器自管 `State{IDLE,DRAG_NODE,DRAG_CONNECTION}`，与 widget 的 PANNING 状态机互斥（widget 仅在 controller 未接管时进 PANNING）。

### D2 拖动节点头=实时 setPosition + 释放时单条命令
连续拖动若每帧 apply 命令会污染 undo 栈（几十条）。方案：
- 按下命中头：记录 `dragTarget=NodeId`、`grabNX=node.x()`、`grabNY=node.y()`、`pressWX=screenToWorld(mouse)`。
- 拖动中：`node.setPosition(grabNX + (curWX-pressWX), grabNY + (curWY-pressWY))` —— **直接改数据，不走命令**，渲染即时跟随。
- 释放：若 `(endX,endY)!=(grabNX,grabNY)` → `undo.apply(new MoveNodeCommand(graph,target,grabNX,grabNY,endX,endY))`（**全量构造**，见 D3）；位移为 0 不 apply（避免空命令占栈）。
- apply 时 execute 再设一次 end（幂等，拖动已设），undo 设回 grabNX/grabNY。redo=execute。语义正确。

### D3 MoveNodeCommand 增加全量构造（最小扩展 TaskC）
新增 `MoveNodeCommand(graph,target,oldX,oldY,newX,newY)`，构造时置 `captured=true`（跳过首次 execute 捕获，直接用传入 old）。原构造 `(graph,target,newX,newY)` 不变（其他场景仍用懒捕获）。两构造共存，不破坏 TaskC 既有测试。

### D4 拖端口连线=双向（输入/输出皆可拖出）
- 按下命中输出端口 → `DRAG_CONNECTION(fromOutput=true)`，源=(该节点, portIndex)。
- 按下命中输入端口 → `DRAG_CONNECTION(fromOutput=false)`，源=(该节点, portIndex)。
- 预览：`ConnectionDrag(fromLayout, portIndex, fromOutput, color)`，renderPending 据方向取 outputPort/inputPort 锚点。
- 释放落点 pick：
  - `fromOutput=true`（从输出拖）→ 落点 pick **输入端口**。
  - `fromOutput=false`（从输入拖）→ 落点 pick **输出端口**。
- 命中后归一化为 `(fromNode,fromOutput,toNode,toInput)`：从输入拖时，落点输出端口是 from，源输入端口是 to。
- `canConnect(fromNode,fromOutput,toNode,toInput)`：OK 或 AUTO_CONVERTED → `undo.apply(new ConnectCommand(...))`（TaskA 单源替换自动处理该 input 已有连线）；INCOMPATIBLE → 静默不连。
- 落空（未命中任何对面端口）→ 不 apply（不断开既有连线；"拖出再松手空白=断开"留作后续打磨，非本任务范围）。

### D5 ConnectionDrag 演化（TaskG→TaskH）
TaskG 的 `ConnectionDrag(fromLayout,outIdx,color)` 仅支持输出端拖。本任务改为 `(NodeLayout fromLayout, int portIndex, boolean fromOutput, int color)`。`NodeGraphWidget.renderPending` 据 `fromOutput` 取 inputPort/outputPort 锚点。这是 TaskG 规格 D4 明确"数据源归 TaskH"的兑现。

### D6 预览线兼容性高亮（释放前的类型校验视觉反馈）
`NodeGraphWidget.renderPending` 在画预览前 pick 当前鼠标落点（据 fromOutput 决定 pick 输入/输出端口）+ `canConnect`：
- 未命中落点 → 类型色半透明（alpha=PREVIEW_ALPHA，当前行为）。
- 命中且 canConnect=OK 或 AUTO_CONVERTED → 类型色满 alpha（0xFF，明显加亮，表示"可接"）。
- 命中且 canConnect=INCOMPATIBLE → 红色半透明 `withAlpha(0xFF0000, 0x80)`（表示"不可接"）。
高亮是即时校验的可视化，不改变释放时的实际校验逻辑（D4）。

### D7 pick 优先级与遍历顺序
按下时遍历 `graph.nodes()`，对每个 `new NodeLayout(node)` 依次检测：`pickOutputPort` → `pickInputPort` → `headerContains`（端口优先于头；端口命中即停）。首个命中的节点/部位接管交互。遍历顺序= nodes() 迭代序（与 TaskF pickHover 一致）。重叠节点不专门处理（上层=列表后；当前从前向后，与渲染视觉略有差异，但节点通常不重叠，记录限制）。

### D8 UndoManager 集成（widget 构造变更）
`NodeGraphWidget` 构造增加 `UndoManager undo` 参数（`Objects.requireNonNull`），新增 `undo()` getter。`NodeGraphScreen` 内部 `new UndoManager()` 传给 widget，并 expose `undo()`（供 TaskI 快捷键 Ctrl+Z / Ctrl+Shift+Z、TaskK 用）。其他嵌入者须自备 UndoManager 传入。

### D9 tooltip 在交互状态下保持
TaskF 的 `renderNodes→pickHover→renderComponentTooltip` 在每次 renderWidget 执行，不依赖交互状态。拖动/拖连线时鼠标在某端口/组件上仍显示其 tooltip（除非鼠标已在拖动起点端口——此时 pickHover 仍命中同端口，tooltip 正常）。无需额外处理。

### D10 右键不处理（归 TaskI）
本任务控制器仅响应 button==0（左键）。button==1（右键）在 controller 返回 false（不接管），widget 也不对其做 PANNING（避免右键拖动画布与 TaskI 右键菜单冲突）。实现方式：`isValidClickButton` 收紧为 `{0,2}`（左+中），右键被默认链视为无效按钮直接 return false。中键(2)继续走 PANNING。

## 交付物

### 新增

  - `NodeInteractionController(NodeGraphWidget widget)`。
  - `boolean onMouseClicked(double mx,double my,int button)`：button!=0 返回 false；命中输出端口→DRAG_CONNECTION(fromOutput=true)+setPending；命中输入端口→DRAG_CONNECTION(fromOutput=false)+setPending；命中头→DRAG_NODE(capture)；否则 false。

## 不变量
1. 控制器 State 同一时刻只在一个状态（IDLE/DRAG_NODE/DRAG_CONNECTION 互斥）。
2. DRAG_NODE/DRAG_CONNECTION 仅由左键(0)进入；中键(2)永远走 widget PANNING；右键(1)不被任何逻辑接管。
3. 拖动节点期间数据实时变更但不入 undo 栈；仅释放时入栈一条 MoveNodeCommand（位移=0 则不入栈）。
4. ConnectCommand 仅在 canConnect∈{OK,AUTO_CONVERTED} 时 apply；INCOMPATIBLE/落空不入栈。
5. 从输出拖→落点必须是输入端口；从输入拖→落点必须是输出端口（D4 归一化）。
6. pending 在 DRAG_CONNECTION 进入时 setPending，在释放/状态离开时 setPending(null)。
7. MoveNodeCommand 全量构造 captured=true 后，execute/redo/undo 行为与原构造首次 execute 后一致（old 固定）。

## 异常
- `MoveNodeCommand` 全量构造：graph/target non-null（Objects.requireNonNull）；坐标无约束。
- controller pick 时节点/端口索引由 NodeLayout 保证（pick 返回 Optional，命中才访问）。
- canConnect 传无效 NodeId → TaskA 抛 IllegalArgumentException（不应发生，落点来自现存节点）。

## 副作用
- 拖动节点：直接改 Node.x/y（数据突变），释放时 apply 命令。
- 连线：apply ConnectCommand → TaskA connect（可能替换该 input 既有连线，被替换连线进 ConnectCommand.oldConnection，undo 可恢复）。
- UndoManager 状态变化触发其 listeners（TaskI 接菜单刷新）。

## 决策记录
1. 实时 setPosition 而非命令流：避免 undo 栈污染 + 保证视觉即时。释放时单条全量命令保证 undo 干净还原到拖动前。
2. MoveNodeCommand 全量构造而非新建独立类：与原命令 90% 重叠，重载构造最小化重复、集中维护。原构造保留以兼容 TaskC 测试。
3. 双向拖连线（输入/输出皆可）：符合"拖输入/输出→拖出线"需求；归一化后统一 ConnectCommand。
4. 落空不断开既有连线：保守默认，避免意外断开；"拔线"交互（拖已连输入到空白=断开）留后续。
5. 预览兼容性高亮：即时校验反馈，提升可用性，不改变释放语义。
6. controller 与 widget 分层：widget 保渲染+pan，controller 管节点交互，职责清晰、可演化（TaskI 选择/菜单可扩展控制器）。
7. isValidClickButton 收紧剔除右键：避免右键被 widget 当 pan 吞掉，为 TaskI 右键菜单留路。
8. pick 顺序端口>头：端口是高频交互目标，头是大区域；端口优先避免头覆盖端口。
9. tooltip 复用 TaskF，不重做：交互不改变悬停语义。
10. UndoManager 经 widget 构造注入：单一来源，TaskI/TaskK 可经 screen/widget 取到同一实例。
