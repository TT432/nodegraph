# TaskC 命令系统与撤销/重做

## 目标
为 NodeGraph 编辑器提供可撤销/重做的命令框架：每个对图的可变操作封装为 `Command`，由 `UndoManager` 维护 undo/redo 双栈。覆盖 TaskA 暴露的全部可变操作。纯 Java 层，仅依赖 TaskA 数据模型 + `java.util` + `net.minecraft.network.chat.Component`（公共侧），不依赖 MC 客户端渲染类。JUnit 可测。

## 前置条件
- TaskA 已完成（commit 45a254a）。复用 NodeGraph / Node / NodeGroup / Connection / InputWidget / NodeDefinition / NodeId / NodeGroupId。
- TaskA 的可变面（精确签名）：
  - `NodeGraph`：`addNode(def,x,y)->Node`、`removeNode(id)`（级联删连线）、`createGroup(h,x,y,w,h)->NodeGroup`、`removeGroup(id)`（清成员 groupId）、`setNodeGroup(nodeId,groupOrNull)`、`connect(...)->Connection`（替换单源）、`disconnect(Connection)`、`inputConnection(node,in)->Optional<Connection>`、`connections()->List<Connection>`。
  - `Node`：`setHeader`、`setPosition`、`widgets()`（可变 List）、`setGroupId`（包级）。
  - `NodeGroup`：`setHeader`、`setPosition`、`setSize`、`setScale`。
  - `InputWidget`：`setCurrentValue`。

## 关键设计决策：节点 id 稳定性

撤销/重做必须保持**对象身份稳定**：
- `connect(fromNode,...)` / `setNodeGroup(nodeId,...)` / 成员派生 `members(group)` 全部以 `NodeId` / `NodeGroupId` 引用节点。
- 若 `AddNodeCommand.redo` 再调 `graph.addNode(def,x,y)`，会分配**新 id**；之前依赖旧 id 的 connection / group 成员关系将永久断裂。
- 因此 `NodeGraph` 必须额外提供**低级恢复原语**，让命令在 redo 时把同一 `Node` / `NodeGroup` / `Connection` 对象原样放回：

### NodeGraph 新增公共 API（恢复原语，**低级语义**）
```java
/** 原样插入节点（不分配 id）。用于命令恢复/反序列化。
 *  校验：id 不冲突。副作用：推进 nextNodeId 至 node.id()+1（若需要）。 */
public void insertNode(Node node);

/** 原样插入节点组（不分配 id）。校验：id 不冲突。副作用：推进 nextGroupId。 */
public void insertGroup(NodeGroup group);

/** 原样添加连线。校验：endpoints 存在；该 input 当前无其它连接（防双源破坏不变量）。
 *  不做类型校验/不做替换 —— 调用方（命令/反序列化）负责语义。 */
public void addConnection(Connection c);
```

> **这不是设计退化**，而是命令系统精确恢复的必要支撑：id 由容器分配的封装在工厂方法 `addNode/createGroup` 上保留；低级 API 仅供命令层与序列化层使用，javadoc 明确标注。

## 范围与交付物

### 包 `com.nodegraph.nodegraph.api.command`

#### `Command`（接口）
```java
public interface Command {
    void execute();                       // 首次执行；必须可重复调用（重做时由懒捕获分支处理）
    void undo();                          // 撤销 execute 的效果
    default void redo() { execute(); }    // 默认重做 = 再 execute；仅当 execute 不可幂等时 override
    String description();                 // UI 显示用（如 "移动节点"）；纯 Java 层返回 String
}
```
**懒捕获契约**：execute 内部据自身状态（如 `savedNode==null`）决定"首次分配"或"用已保存对象 restore"。这样默认 `redo()` 复用 execute 即可正确工作。

#### `UndoManager`
- 双栈：`Deque<Command> undoStack`（已执行）、`redoStack`（已撤销）。
- `void apply(Command c)`：`c.execute()` → 推入 undoStack → **清空 redoStack** → 裁剪至 `depthLimit` → fireChanged。
- `boolean undo()`：空栈返回 false；否则 pop→`undo()`→push redo→fireChanged→true。
- `boolean redo()`：空栈返回 false；否则 pop→`redo()`→push undo→fireChanged→true。
- `boolean canUndo()` / `boolean canRedo()`。
- `void clear()`：清空两栈。
- `int undoDepth()` / `int redoDepth()`（测试用）。
- 监听：`addListener(Runnable)` / `removeListener(Runnable)`；每次 apply/undo/redo/clear 后触发，UI 据此刷新按钮状态。
- `depthLimit`（默认 100）：undoStack 超过则丢弃栈底（最旧命令不可撤销）。

#### `CompositeCommand implements Command`
- 持有 `List<Command> children`（构造时传入，不可变）。
- execute：顺序执行子命令。
- undo：**逆序** undo 子命令。
- redo：默认 = execute（顺序）。
- 用于"右键组合节点入组"（CreateGroup + N×SetNodeGroup）、"剪贴板粘贴多节点"（TaskD 复用）。

#### 具体命令（每个独立文件）

| 命令 | execute | undo | 快照内容 |
|---|---|---|---|
| `AddNodeCommand` | 首次：`graph.addNode(def,x,y)` 保存返回 Node；重做：`graph.insertNode(savedNode, cascadedConns)` | 捕获图中引用该 node 的全部 connection → `cascadedConns`；`graph.removeNode(id)` | Node + 级联 connections |
| `RemoveNodeCommand` | 首次：捕获 node + 引用它的 connections；`removeNode`。重做：`removeNode` | `graph.insertNode(savedNode)`；`graph.addConnection(each saved)` | Node + 级联 connections |
| `MoveNodeCommand` | 捕获 oldX/oldY（首次）；`node.setPosition(newX,newY)` | `node.setPosition(oldX,oldY)` | old position |
| `SetNodeHeaderCommand` | 捕获 oldHeader；`node.setHeader(newHeader)` | `node.setHeader(oldHeader)` | old header |
| `SetWidgetValueCommand` | 捕获 oldValue；`widget.setCurrentValue(newValue)` | `widget.setCurrentValue(oldValue)` | old value |
| `ConnectCommand` | 每次重新捕获 `oldConn = graph.inputConnection(toNode,toInput).orElse(null)`；`graph.connect(from,fo,to,ti)` | 按四元组查找并 disconnect 当前连线；若 `oldConn!=null` 则 `addConnection(oldConn)` | 上次 execute 时的旧 connection |
| `DisconnectCommand` | 首次：保存要断开的 `conn`；`disconnect(conn)`。重做：`disconnect(conn)` | `graph.addConnection(conn)` | connection |
| `CreateGroupCommand` | 首次：`graph.createGroup(h,x,y,w,h)` 保存返回；重做：`graph.insertGroup(savedGroup)` | `graph.removeGroup(id)`（成员关系如何？见下） | NodeGroup |
| `RemoveGroupCommand` | 首次：保存 group + 各成员的 NodeId 列表；`removeGroup`。重做：`removeGroup` | `graph.insertGroup(savedGroup)`；对每个成员 `graph.setNodeGroup(memberId, savedGroup.id())` | NodeGroup + 成员 id 列表 |
| `SetGroupHeaderCommand` | 捕获 oldHeader；`group.setHeader(new)` | `group.setHeader(old)` | old header |
| `SetGroupTransformCommand` | 捕获 oldPos/oldSize/oldScale；设新值 | 恢复旧值 | old transform |
| `SetNodeGroupCommand` | 捕获 oldGroupId；`graph.setNodeGroup(nodeId, newGroupId)` | `graph.setNodeGroup(nodeId, oldGroupId)` | old group id |

**关于 CreateGroupCommand 的成员恢复**：
- 首次 execute 时图刚创建，无成员。
- 用户随后通过 SetNodeGroupCommand 把节点加入——这些是独立的命令，会独立 undo。
- 若 CreateGroupCommand 自身 undo（removeGroup），TaskA 的 removeGroup 会清空成员的 groupId。
- 此后 CreateGroupCommand.redo（insertGroup）只重新插入空 group；成员的 groupId 已被清掉，**无法靠 SetNodeGroupCommand 的 undo 恢复**（因为 SetNodeGroupCommand 是栈中更上层的命令，还未 undo 到它）。
- **决策**：CreateGroupCommand 在 execute 时快照"当前图状态"无意义（首次无成员；redo 时也无需恢复成员，因为 SetNodeGroupCommand 的 undo 序列会处理）。但 CreateGroupCommand.undo 调用 `removeGroup` 会**破坏后续 SetNodeGroupCommand.undo 时的 group 存在性**（group 不存在了，setNodeGroup 抛）。
- **解决**：CreateGroupCommand.undo 不调 `removeGroup`，而是用低级路径：先清空成员的 groupId（手动），再从 groups map 移除 group（**需 NodeGraph 暴露 `removeGroupOnly(id)` 或让 insertGroup/removeGroup 的低级版可见**）。
- **简化方案**：让 CreateGroupCommand.undo 调用一个新低级 API `NodeGraph.removeGroupRaw(id)`（仅移除 group 实体，不动成员），并配合手动管理成员。但成员关系由成员自身的 SetNodeGroupCommand 管理——CreateGroupCommand 不应越权。
- **最终决策**：CreateGroupCommand 不自动撤销成员关系。其 undo 仅从图中移除 group 实体（保留成员的 groupId 字段，但成员的 groupId 此时指向已不存在的 group——下次 setNodeGroup 校验时才暴露问题）。
- 为避免"悬空 groupId"，**提供 `NodeGraph.removeGroupRaw(NodeGroupId)`**：仅删除 group 实体，不动成员。CreateGroupCommand 用它 undo。SetNodeGroupCommand 的 oldGroupId 若指向已 removeGroupRaw 的 group，undo 时 setNodeGroup(nodeId, oldGroupId) 会触发 `group(oldGroupId)` 校验失败。
- **真正的解决**：SetNodeGroupCommand.execute 时**不校验** group 存在性；只 setGroupId。但 TaskA 的 `setNodeGroup` 现在校验。改 TaskA？不在 TaskC 范围。
- **折衷决策**：CreateGroupCommand.undo 用普通 `removeGroup`（会清成员 groupId）。这意味着如果用户：CreateGroup → SetNodeGroup(join) → undo CreateGroup → redo CreateGroup，中间的 join 状态会丢失。但这是合理的"乱序 undo"边缘情况，UndoManager 的线性 undo 不允许跳过中间命令，所以实际不会发生。**记录此约束在 javadoc**：命令依赖 UndoManager 的线性 undo 序列。

> 上述 CreateGroupCommand/RemoveGroupCommand 的成员关系处理是本任务最微妙的点。结论：**信任 UndoManager 的线性 undo 序列**，命令不防御乱序调用。

## 不变量
1. UndoManager 的 undoStack 中命令均处于"已 execute"状态；redoStack 中均处于"已 undo"状态。
2. apply 新命令时清空 redoStack（用户分支切换）。
3. 命令的 execute↔undo 必须可重复（undo→redo→undo→redo…）且每次恢复到等价状态。
4. 创建型命令（AddNode/CreateGroup）的 redo 复用 execute 首次保存的对象身份（id 稳定）。
5. NodeGraph 的低级恢复原语（insertNode/insertGroup/addConnection）**不破坏** TaskA 既有不变量（id 唯一、input 单源）——它们做最小校验。
6. CompositeCommand 的子命令 undo 逆序，保证多命令原子的可逆性。

## 异常行为
- `UndoManager.apply` 时若 `c.execute()` 抛 → 不入栈，异常传播；redoStack 不变（用户应处理）。
- `undo`/`redo` 时若命令抛 → 栈状态可能不一致（已 pop 未 push）；异常传播。命令契约要求 execute/undo 不抛（除非数据真损坏）。
- 空栈 undo/redo 返回 false，不抛。
- `insertNode` id 冲突 → `IllegalStateException`。
- `addConnection` endpoints 不存在 → `IllegalArgumentException`；该 input 已有**其它**连接 → `IllegalStateException`。

## 副作用
- 命令修改 NodeGraph（这是其职责）。
- UndoManager 维护双栈，触发监听器。
- 无 IO、无持久化。

## 决策记录
1. **低级恢复原语**（insertNode/insertGroup/addConnection）：必须新增，保证 id/对象身份稳定。封装边界在工厂方法（addNode/createGroup/connect）；低级 API 文档标注"命令/序列化专用"。
2. **懒捕获模式**：execute 内部判断首次/重做，使 `redo()` 默认 = execute。减少样板，单一入口。
3. **每次 execute 重新捕获 oldConn（ConnectCommand）**：线性 undo 序列保证 oldConn 始终是"上次 execute 留下的"；鲁棒于"undo 后 input 状态被同序列其他命令改变"的合法情况（虽然理论上不会发生，但更安全）。
4. **不引入 transaction / 合并机制**：拖动连发的 MoveNodeCommand 由 UI 层（TaskH）决定何时 apply（如鼠标释放时一次性 apply 最终位置）。TaskC 只提供原子 Command。
5. **信任 UndoManager 线性 undo**：命令不防御乱序调用（如 CreateGroup.undo 后 SetNodeGroup.redo），文档明确。
6. **Command.description() 返回 String**：纯 Java 层不依赖 Component；UI 层（TaskI）自行 i18n。
7. **CompositeCommand 子命令 execute 顺序、undo 逆序**：标准 macro 模式。
8. **depthLimit=100**：防止长会话栈无限增长；最旧命令被丢弃（不可撤销）。

## 验证（JUnit 5）
`src/test/java/com/nodegraph/nodegraph/api/command/TestCommands.java` + `TestUndoManager.java`：

### TestCommands（每个命令的 execute/undo/redo 可重复 + 不变量）
1. AddNodeCommand：execute→存在；undo→不存在；redo→存在；**多次 undo/redo 后 NodeId 不变**。
2. AddNodeCommand 级联：add A、add B、connect A→B；undo add A（自动带走 connect）；redo add A（connect 自动回来，仍引用 A 的稳定 id）。
3. RemoveNodeCommand：add A、connect A→B；remove A（级联）；undo → A 与 connect 都恢复；redo → 都消失。
4. MoveNodeCommand：位置变更、undo、redo、可重复。
5. SetNodeHeaderCommand：header 变更、undo、redo。
6. SetWidgetValueCommand：值变更、undo、redo。
7. ConnectCommand 替换语义：初始 X→I；connect B→I（替换 X）；undo（恢复 X）；redo（再替换）。
8. DisconnectCommand：断开、undo 恢复、redo。
9. CreateGroupCommand：创建、undo（图无 group）、redo（id 稳定）。
10. RemoveGroupCommand：创建组、加入 2 节点、移除组（成员 groupId 清空）、undo（组与成员关系都恢复）、redo。
11. SetGroupHeaderCommand / SetGroupTransformCommand：变更、undo、redo。
12. SetNodeGroupCommand：加入/退出组、undo、redo。
13. CompositeCommand：CreateGroup + 2×SetNodeGroup；execute 全生效；undo 全恢复；redo 全生效。

### TestUndoManager
14. apply → canUndo true；undo → canUndo false、canRedo true；redo → canUndo true、canRedo false。
15. apply A、apply B、undo、undo、redo、redo：正确顺序。
