# TaskN 规划：连接生命周期事件 + 通知机制

## 1. 任务目标
为 `NodeGraph` 中 `Connection` 的生命周期变更提供**细粒度、可订阅**的事件通知机制，
使外部消费者（求值缓存失效、UI 重渲染、外部同步、用户回调）能响应连接的创建 / 移除，
而无需轮询 `graph.connections()`。

## 2. 背景与现状
- 连接的变更入口全部集中在 `NodeGraph`：
  - `connect(...)`：校验 → 替换语义（先 `disconnect(old)` 再 `add(new)`）。
  - `disconnect(Connection)`：`List.remove`。
  - `addConnection(Connection)`：低层级追加（命令 undo / 序列化恢复），幂等。
  - `removeNode(NodeId)`：`removeIf` 级联删除引用该节点的所有连接。
- 命令系统（`ConnectCommand` / `DisconnectCommand`）只调用上述公共方法，不直接操作 `connections`。
- 既有的通知模式 = `UndoManager`：`List<Runnable>` + `addListener/removeListener` + `fireChanged()`，
  属**粗粒度**「状态变了」通知。本 Task 提供**细粒度**（带 `Connection` 上下文）通知，二者并存、互不替代。

## 3. 设计决策
- **D1 事件类型枚举**：`CREATED` / `REMOVED`。
  替换语义 = `REMOVED(old)` + `CREATED(new)` 两个独立事件（由 `connect` 内部先 `disconnect(old)`
  天然产生，监听者无需特殊处理）。
- **D2 不可变事件对象**：`ConnectionEvent { Type type; Connection connection; }`。
  未来若需携带「原因（cascade/undo/user）」可扩字段，当前不引入（YAGNI）。
- **D3 函数式监听者接口**：`ConnectionListener { void onConnectionEvent(ConnectionEvent e); }`，
  而非 `Runnable`——因为需携带 `Connection` 上下文。
- **D4 图级注册**：监听者挂在 `NodeGraph`（连接的唯一管理者），
  与 `UndoManager` 一致；不做端口级 / 单 Connection 级注册（过度细分，与既有模式不符）。
- **D5 事后通知（post-notification, informational）**：变更成功后才触发，**不可否决**。
  事前校验仍由 `canConnect` / 端点存在性检查承担。与 `UndoManager.fireChanged()` 一致。
- **D6 触发点覆盖全部变更路径**：
  | 方法 | 触发 |
  |------|------|
  | `connect(...)` | 替换时经内部 `disconnect(old)` 触发 `REMOVED(old)`；新建后 `CREATED(new)` |
  | `disconnect(c)` | 仅当 `List.remove` 真正移除时触发 `REMOVED(c)` |
  | `addConnection(c)` | 仅当实际新增（非幂等命中）时触发 `CREATED(c)` |
  | `removeNode(id)` | 级联删除的**每一条**触发 `REMOVED`（监听者自行去抖） |
- **D7 重入安全**：触发时遍历监听者列表的**快照**（copy-on-iterate），防 `ConcurrentModificationException`。
  监听者回调中对 `addConnectionListener/removeConnectionListener` 的修改不影响当前本轮派发。
- **D8 异常隔离**：单个监听者抛 `RuntimeException` 不应中断其他监听者，也不应破坏图变更的最终状态。
  派发循环 `try/catch`，异常打印到日志（`System.err`，保持纯 Java 层零 MC 依赖）后继续。
- **D9 纯 Java 层**：新类放 `api.model`，不引入 MC 依赖，可 JUnit 单测。

## 4. 假设与边界（决策记录）
- **不实现特定消费者**：本 Task 仅交付事件基础设施。让 `Evaluator` 订阅以失效缓存、
  或让 UI 订阅以重渲染，均**不在本 Task**（留作后续 Task，避免耦合与范围蔓延）。
- **不做批量 / 事务性通知**：`removeNode` 级联删除多条时每条单独触发；
  监听者可自行累积去抖。符合最小接口原则与 `UndoManager` 既有风格。
- **不做事件总线 / 跨图分发**：单 `NodeGraph` 实例 → 其监听者列表；无全局静态总线。
- **不引入弱引用监听者**：与 `UndoManager` 一致用强引用，调用方负责 `removeListener`。

## 5. 交付物
- `api/model/ConnectionEvent.java`（不可变值对象 + `Type` 枚举内嵌或独立）
- `api/model/ConnectionListener.java`（函数式接口）
- `NodeGraph`：`addConnectionListener / removeConnectionListener` + 私有 `fireConnectionEvent`
  + 在 4 个变更点（`connect` / `disconnect` / `addConnection` / `removeNode`）插入触发。
- `TestConnectionEvents.java`：覆盖创建、断开、替换、级联删除、幂等 addConnection、
  多监听者、重入（回调中注销）、异常隔离。

## 6. 不变量
1. `CREATED` 事件中的 `Connection` 必须在触发瞬间**已存在**于 `graph.connections()`。
2. `REMOVED` 事件中的 `Connection` 必须在触发瞬间**已不存在**于 `graph.connections()`。
3. 一次成功 `connect`：若发生替换，事件序列严格为 `REMOVED(old)` → `CREATED(new)`。
4. 幂等 `addConnection`（连接已存在）：不触发任何事件。
5. 监听者回调抛异常不影响后续监听者被通知，也不回滚已完成的图变更。
6. 触发期间对监听者列表的结构性修改不影响本轮派发顺序。

## 7. 验证
- `gradlew test`：新增 `TestConnectionEvents` 全过，既有测试不回归。
- 人工审查：触发点与变更点一一对应（见 D6 表）。

## 8. 依赖与顺序
- 依赖 TaskA（数据模型已就绪）。
- 与既有 `UndoManager` 监听者机制并存，互不依赖。
