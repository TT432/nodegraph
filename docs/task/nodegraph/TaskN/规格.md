# TaskN 规格说明：连接生命周期事件 + 通知机制

## 背景
`NodeGraph` 此前仅通过 `UndoManager` 提供**粗粒度**「状态变了」通知（`List<Runnable>`）。
连接（`Connection`）的创建 / 移除没有可订阅的细粒度事件，外部消费者（求值缓存失效、
UI 重渲染、外部同步、用户回调）只能轮询 `graph.connections()` 做差比对，代价高且易错。

## 目标
为 `Connection` 的生命周期提供细粒度、可订阅的事件通知基础设施。

## 前置（复用）
- TaskA 数据模型（`NodeGraph` / `Connection` / `ConnectResult`）。
- 既有 `UndoManager` 的 `List<Runnable>` + `addListener/removeListener` + `fireChanged()` 风格作为模式参照。

## 关键设计决策
- **D1 事件类型**：`ConnectionEvent.Kind { CREATED, REMOVED }`。替换语义（`connect` 对已连接输入再连）
  天然映射为 `REMOVED(old)` + `CREATED(new)` 两个独立事件——由 `connect` 内部先调用
  `disconnect(old)` 自动产生，监听者无需特判。
- **D2 不可变事件对象**：`ConnectionEvent { Kind kind; Connection connection; }`，两字段均非空。
- **D3 函数式监听者接口**：`ConnectionListener.onConnectionEvent(ConnectionEvent)`（`@FunctionalInterface`），
  携带 `Connection` 上下文（非 `Runnable`）。
- **D4 图级注册**：`NodeGraph.addConnectionListener / removeConnectionListener`，
  与 `UndoManager` 一致；不做端口级 / 单 Connection 级注册。
- **D5 事后通知、不可否决**：变更成功后才触发；事前校验仍由 `canConnect` / 端点存在性检查承担。
- **D6 触发点**：
  | 方法 | 行为 |
  |------|------|
  | `connect(...)` | 经内部 `disconnect(old)` 触发 `REMOVED(old)`；新建后触发 `CREATED(new)` |
  | `disconnect(c)` | 仅当 `List.remove` 真正移除（返回 true）时触发 `REMOVED(c)`；幂等无事件 |
  | `addConnection(c)` | 仅当实际新增（非幂等命中 `contains`）时触发 `CREATED(c)` |
  | `removeNode(id)` | 收集级联受害者后逐条 `remove` + 触发 `REMOVED`（原 `removeIf` 改为先收集再逐条移除） |
- **D7 重入安全**：派发遍历监听者列表的快照（`new ArrayList<>(connectionListeners)`），
  回调中对列表的 add/remove 不影响本轮派发。
- **D8 异常隔离**：派发循环 `try/catch RuntimeException`，异常打印 `System.err` 后继续，
  单个坏监听者不影响其他监听者，也不回滚已完成的图变更。
- **D9 纯 Java 层**：新类位于 `api.model`，零 MC 依赖，可 JUnit 单测。

## 不在本 Task（边界）
- 不实现任何具体消费者（Evaluator 缓存订阅、UI 重渲染订阅等留作后续）。
- 不引入批量 / 事务性通知（`removeNode` 级联逐条触发，消费者自行去抖）。
- 不做事件总线 / 跨图分发 / 弱引用监听者。
- 不引入「原因（cascade/undo/user）」字段（YAGNI，需要时再扩）。

## 交付物
- `api/model/ConnectionEvent.java`（不可变值对象 + 内嵌 `Kind` 枚举）
- `api/model/ConnectionListener.java`（函数式接口）
- `NodeGraph`：`connectionListeners` 字段 + `addConnectionListener` / `removeConnectionListener`
  + 私有 `fireConnectionEvent(Kind, Connection)`；4 个变更点插入触发。
- `TestConnectionEvents`：14 个用例覆盖全部路径。

## 不变量
1. `CREATED` 触发瞬间，`connection` 已存在于 `graph.connections()`。
2. `REMOVED` 触发瞬间，`connection` 已不存在于 `graph.connections()`。
3. 一次成功 `connect` 若发生替换，事件序列严格为 `REMOVED(old)` → `CREATED(new)`。
4. 幂等 `addConnection`（连接已存在）→ 无事件。
5. `disconnect` 未持有的连接 → 无事件（幂等）。
6. 监听者回调抛 `RuntimeException` 不影响其他监听者被通知，不回滚图变更。
7. 派发期间对监听者列表的结构性修改不影响本轮派发顺序与覆盖范围。

## 异常行为
- `addConnectionListener(null)` → `NullPointerException`。
- 监听者回调抛任意 `RuntimeException` → 被 `fireConnectionEvent` 捕获、打印 `System.err`、吞掉。

## 副作用
- 仅内存对象图；监听者同步回调在同一线程、变更之后触发。
- 无 IO、无持久化、无全局静态状态。

## 验证
- `gradlew test`：`TestConnectionEvents`（14 用例）全过；既有套件无回归（合计 146 tests / 0 failures）。
- 编译产物含 `ConnectionEvent.class`、`ConnectionEvent$Kind.class`、`ConnectionListener.class`、

## 决策记录
- 事件对象 vs 双回调：选事件对象（携带 `Kind` 枚举），未来扩字段（如「原因」）无需改签名。
- 是否做 `REPLACED` 第三种 Kind：否——替换 = remove + create 两步，更通用、更易组合。
- 异常打印走 `System.err`：保持纯 Java 层零 MC 依赖（不引入 `Log4j` / Forge logger）。
- `removeNode` 由 `removeIf` 改为先收集再逐条 `remove` + fire：`removeIf` 无法在移除时回调，必须显式化。
